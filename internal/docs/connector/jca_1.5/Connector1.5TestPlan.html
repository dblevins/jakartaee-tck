J2EE Compatibility Test Plan (CTS) for Connector 1.5 Plan
 
Gursharan Singh
last updated  08/23/2002
Introduction
This specification documents the Connector 1.5 high level Compatiblity CTS test plan for the Connector 1.5 specification. Compatibility test cases are derived from the requirements of these engineering specifications. The reference implementation will validate that the specifications can be implemented, will provide a reference point for the development community, and will be used for developing Compatibility Tests, which leads to a brand. It is a requirement of the compatibility program that all tests cases be passed before a brand can be applied.

 

Goals

Provide Connector 1.5 compatibility tests for J2EE version 1.4. This will include writing tests for system level contracts between a J2EE 1.4 Application Server and a resource adapter which satisfies J2EE Connector 1.5 specifications. The contracts between J2EE 1.4 Application Server and resource adapter are:
 

  • Work management contract: This allows a resource adapter to submit Work instances to be executed by an application server.
  • Life cycle management contract: This allows the runtime and deployment behaviour of the resource adapter into application server.
  • Message Inflow: This provides a mechanism for EIS to the use resource adapter to dispatch calls to MDB.
  • Transaction Inflow : This provides a contract to import external transaction into the application server.
Non-Goals
  •          Performance
  •          Scalability
  •          Ease of use
  •          Non-required areas of the specification.
  •          The capability to certify any Resource Adapter Connector 1.5 specification.
Types of Compatibility Tests
 
There are 3 types of tests that make up the Compatibility Test Suite, these include:
  • Signature Tests - Checks that all of the required API signatures are available for an implementation, no supersets or subsets.
  • API Tests - Verifies that all of the required classes and methods behave according to the specification on a method by method basis.
  • End to End Tests - A new type of compatibility test which is new to the J2EE platform. These type of tests span across n-tiers of the J2EE Platform.The CTS Connectors tests will be end to end tests, which allow the testing of life cycles, state transitions, and general behaviour.
CTS Tragated High Level Areas

The following areas have been identified as requirements of the Connector specification and are targeted for CTS test development. Please see the Connector specification for
details on these requirements.
 

  • The Connector defined SPI (Server Provider Interface) for the Life Cycle contracts. Described in Connector 1.5 Specification Chapter 5.
  •  The Connector defined SPI (Server Provider Interface) for the Work Management contracts. Described in Connector 1.5 Specification Chapter 10.
  • The Connector defined SPI (Server Provider Interface) for the Transaction Inflow contracts. Described in Connector 1.5 Specification Chapter 12.
  • The Connector defined SPI (Server Provider Interface) for the Message Inflow contracts. Described in Connector 1.5 Specification Chapter 14.
  • The Connector defined SPI (Server Provider Interface) for the Packaging and Deployment contracts. Described in Connector 1.5 Specification Chapter 17.


CTS Non-Targeted High Level Areas
 

  •  The Connector 1.5 specification recommends, but does not require, that a resource adapter support CCI.  As described earlier, the CTS is a "standard bar" applied to all     Licensees.  CTS tests are derived from the requirements of the specifications.  The CTS will not contain any tests for the CCI.
  •  The CTS Connectors tests will not be written for any resource adapter.  A specific, known Resource Adapter (described later in this document) will be deployed on a licensees J2EE server in order to test the J2EE application servers' Connector implementation.  A TCK for Resource Adapters is beyond the scope of this specification.


Test Approach

The purpose of the CTS connector is to insure that J2EE 1.4 Application server has fullfiled all the Connector 1.5 specification which allows any Connector 1.5 resource adpater to be deployed and function correctly.

It will be necessary to have a resource adapter which fullfils the Connector 1.5 specification to be deployed on J2EE 1.4 server. Previouly, for Connector 1.0, CTS took the approach to built its own CTS resource adapter and write tests on top of it. For Connector 1.5 three resource adapter will be built by the CTS team with will use CTS EIS for transaction support and on that assertions will be built on the following areas
 

  • Life Cycle Contract: The following areas will be coverd
      • ResourceAdapter runtime behavior
      • ResourceAdapter association with ManagedConnectionFactory and ActivationSpec
  • WorkManager Contract: The following areas will be coverd
      • To ensure that WorkManager is correctly implemented by J2EE 1.4 Application server
      • Work Object from resource adapter can be passed to WorkManager.
  • Transaction Inflow contract:  The following areas will be covered
      • Message delivery is transacted or not.
      • XAResource being used by the Transaction Manager.
  • Message Inflow Contract:
      • Messages can be dispactched to an non transaction requried MDB.
      • Messages can be dispactched to an transaction requried MDB.
  • Deployment Tests.
      • Embeded resource adapter can be deployed correctly and can be used by the application it is deployed with.
  • Set of Regression tests for Connector 1.0 outbound contracts.
      • A set of connection tests to verify it the Connector 1.0 Resource adapter can work on J2EE 1.4 server.
     Furthermore, the test strategies will evolve with discussions with CTS, RI  team and as the Connector 1.5 spec progresses.
